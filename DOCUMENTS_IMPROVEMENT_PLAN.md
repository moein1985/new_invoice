# برنامه بهبود بخش اسناد (Documents)

تاریخ: 2026-04-04

## 1) هدف این سند
این سند برای شروع پیاده‌سازی بهبودهای بخش اسناد تهیه شده و شامل موارد زیر است:
1. تحلیل مشکلات فعلی (با تمرکز روی PDF و Pagination)
2. راهکار پیشنهادی فنی و UX
3. نقشه فازبندی اجرا
4. معیارهای پذیرش (Acceptance Criteria)
5. تست‌ها و ریسک‌ها

---

## 2) جمع‌بندی وضعیت فعلی (بررسی انجام‌شده)

### 2.1 مشکل PDF در چند صفحه
مشاهده شده خروجی PDF وقتی تعداد ردیف‌ها زیاد می‌شود، ردیف‌ها را وسط صفحه می‌شکند و تراز صفحه اول/دوم به هم می‌ریزد.

**علت اصلی:**
روش فعلی در `lib/services/pdf-export-html.ts` کل صفحه را با `html2canvas` به یک تصویر تبدیل می‌کند و سپس تصویر را با offset منفی روی صفحات بعدی می‌اندازد. این روش «ساختار ردیف جدول» را نمی‌شناسد و برای همین شکست ردیف کنترل‌پذیر نیست.

### 2.2 مشکل نمایش فقط 50 سند
در صفحه لیست اسناد، داده‌ها همیشه با `page=1` و `limit=50` گرفته می‌شوند و هیچ مکانیزم صفحه‌بندی واقعی در UI وجود ندارد.

**علت اصلی:**
1. `app/documents/page.tsx` همیشه صفحه اول را می‌گیرد.
2. Pagination component در این صفحه استفاده نشده.
3. جستجو/فیلتر روی همان 50 رکورد انجام می‌شود، نه روی کل دیتابیس.

### 2.3 تایید دیتای واقعی سرور
در سرور production تعداد اسناد در دیتابیس برابر 184 است (جدول `documents`).
این اختلاف، گزارش کاربر را تایید می‌کند.

## 3) وضعیت فعلی کد و نقاط درگیر

### 3.1 فایل‌های کلیدی
1. `app/documents/page.tsx` (لیست اسناد)
2. `server/api/routers/document.ts` (API اسناد)
3. `lib/services/pdf-export-html.ts` (PDF فعلی مورد استفاده)
4. `app/documents/[id]/page.tsx` (فراخوانی دکمه PDF)
5. `components/ui/pagination.tsx` (کامپوننت آماده Pagination)

### 3.2 نکات فنی مهم
1. Backend قابلیت pagination دارد، اما فرانت درست از آن استفاده نمی‌کند.
2. API لیست اسناد `items` را هم برمی‌گرداند که payload را سنگین می‌کند.
3. چند سرویس PDF موازی در پروژه وجود دارد (`pdf-export.ts`, `pdf-export-v2.ts`, `pdf-export-html.ts`) اما مسیر اصلی از `pdf-export-html.ts` استفاده می‌کند.
4. در `app/documents/page.tsx` یک state به نام `selectedType` تعریف شده ولی عملا در UI استفاده کامل ندارد.

## 4) راهکار پیشنهادی برای مشکل PDF

## 4.1 تصمیم معماری پیشنهادی
پیشنهاد: **خروجی PDF ساختاریافته (table-aware)** به جای image slicing.

### گزینه پیشنهادی (Recommended)
استفاده از `pdfmake` به عنوان مسیر اصلی PDF برای اسناد:
1. جدول با `headerRows: 1` و تکرار هدر در صفحه بعد
2. جلوگیری از شکستن نامناسب ردیف‌ها با `dontBreakRows: true`
3. کنترل دقیق عرض ستون‌ها، فونت، spacing و alignment
4. امکان توسعه تمیز برای Footer/Header شماره صفحه

### چرا این گزینه بهتر است؟
1. ردیف‌ها منطقی و ساختاری شکسته می‌شوند، نه تصویری
2. برای اسناد طولانی پایدارتر و قابل پیش‌بینی‌تر است
3. نگهداری و تست‌پذیری بهتر نسبت به html2canvas

## 4.2 تغییرات اجرایی پیشنهادی
1. مسیر اصلی دکمه PDF در `app/documents/[id]/page.tsx` از `generateDocumentPDFFromHTML` به سرویس ساختاریافته واحد تغییر کند.
2. سرویس‌های PDF موازی تعیین‌تکلیف شوند:
   - یک سرویس Canonical (فعال)
   - سرویس‌های قدیمی یا deprecated شوند یا حذف کنترل‌شده بگیرند
3. فونت فارسی برای PDF به صورت پایدار (ترجیحا local asset) تعریف شود.
4. در صورت نیاز به Export سریع تصویری، گزینه ثانویه (Legacy) با هشدار «کیفیت/شکست ردیف تضمین نمی‌شود» نگه داشته شود.

## 4.3 معیار پذیرش PDF
1. سند با 80 تا 200 ردیف بدون شکست ناقص ردیف بین صفحات تولید شود.
2. هدر جدول در تمام صفحات تکرار شود.
3. ستون‌ها در تمام صفحات تراز ثابت داشته باشند.
4. در داده فارسی، متن ناخوانا یا جابجایی RTL/LTR رخ ندهد.
5. زمان تولید PDF روی سند 150 ردیفی قابل قبول باشد (مثلا < 3-5 ثانیه روی سیستم معمولی).

## 5) راهکار پیشنهادی برای مشکل Pagination و مقیاس‌پذیری لیست اسناد

## 5.1 تصمیم معماری پیشنهادی
**Pagination و filtering باید server-driven باشد**.

### اصل راهکار
1. فرانت state صفحه، سایز صفحه، فیلتر نوع، سرچ و... را نگه دارد.
2. API لیست بر اساس همین state کوئری بگیرد.
3. خروجی API شامل `data + meta(page,total,totalPages,limit)` باشد.
4. `Pagination` component در صفحه اسناد فعال شود.

## 5.2 تغییرات Backend پیشنهادی
در `server/api/routers/document.ts`:
1. اضافه شدن پارامترهای اختیاری `search`, `deepSearch`, `types[]`, `dateFrom`, `dateTo` (در صورت نیاز)
2. جستجوی عمیق روی فیلدهای سند و آیتم‌ها به شکل server-side
3. کاهش payload لیست:
   - در list، `items` کامل برنگردد مگر واقعا لازم باشد
   - در صورت نیاز فقط `_count` یا خلاصه آیتم‌ها
4. مرتب‌سازی قابل تنظیم (default همان createdAt desc)

## 5.3 تغییرات Frontend پیشنهادی
در `app/documents/page.tsx`:
1. اضافه کردن state های زیر:
   - `page`
   - `limit`
   - `search`
   - `types`
2. اتصال این state ها به query `trpc.document.list`
3. استفاده از `components/ui/pagination.tsx`
4. نمایش دقیق شمارنده:
   - تعداد کل نتایج مطابق فیلتر
   - تعداد رکورد همین صفحه
5. Sync با URL Query String برای قابل اشتراک‌گذاری بودن لینک جستجو/فیلتر

## 5.4 معیار پذیرش Pagination
1. اگر total=184 باشد، کاربر بتواند همه صفحات را ببیند.
2. تغییر صفحه بدون reset شدن فیلترها انجام شود.
3. جستجو/فیلتر روی کل دیتاست اعمال شود، نه فقط page جاری.
4. سرعت پاسخ لیست روی دیتای بزرگ همچنان قابل قبول بماند.

## 6) پیشنهادهای UX/UI برای بخش اسناد

## 6.1 لیست اسناد
1. نوار فیلتر sticky در بالای جدول
2. قابلیت انتخاب سایز صفحه: 20 / 50 / 100
3. نمایش active filters به صورت chip با امکان حذف سریع
4. دکمه «پاک‌کردن همه فیلترها»
5. Export با دو گزینه:
   - Export current page
   - Export all filtered (server-side)

## 6.2 صفحه جزئیات سند
1. گروه‌بندی دکمه‌ها: Primary actions / Secondary actions
2. تاییدیه حذف/تبدیل به شکل modal استاندارد به جای `confirm/alert`
3. اضافه شدن وضعیت «در حال تولید PDF» با progress/disabled state

## 6.3 صفحه ایجاد/ویرایش سند
1. انتخاب مشتری با جستجوی server-side (نه limit=100 ثابت)
2. Shortcut برای افزودن ردیف جدید (مثل Enter در آخرین ردیف)
3. هشدار معتبرسازی inline کنار هر فیلد
4. sticky summary panel (جمع کل، تخفیف، نهایی)

---

## 7) نکات عملکردی (Performance) پیشنهادی

1. کم‌کردن payload list (حذف `items` کامل از لیست)
2. ایندکس‌گذاری هدفمند در دیتابیس روی فیلدهای پرترافیک:
   - `createdAt`
   - `documentType`
   - `customerId`
   - (در صورت نیاز) `approvalStatus`
3. جلوگیری از N+1 در query های مرتبط
4. اعمال debounce برای سرچ در UI

## 8) برنامه فازبندی اجرای پیشنهادی

## فاز 0: آماده‌سازی (نیم روز)
1. انتخاب مسیر canonical برای PDF
2. تعریف API contract نهایی برای list filters
3. آماده‌سازی test data پرحجم

## فاز 1: Pagination و فیلتر server-side (1 تا 2 روز)
1. اصلاح API لیست اسناد
2. اتصال UI صفحه اسناد به pagination واقعی
3. اتصال کامل فیلترها به backend
4. اضافه کردن Pagination component

## فاز 2: بازطراحی PDF (1 تا 2 روز)
1. جایگزینی مسیر تصویری با مسیر ساختاریافته
2. پیاده‌سازی table header repeat + row-safe pagination
3. کنترل فونت و RTL

## فاز 3: UX polishing (1 روز)
1. بهبود فیلترها و نمایش شمارنده‌ها
2. modal/action feedback استاندارد
3. بهبود تجربه export

## فاز 4: تست و پایش (نیم تا یک روز)
1. Unit test برای query builder و mapping
2. E2E برای لیست اسناد و صفحه‌بندی
3. سناریوی PDF چندصفحه‌ای با داده زیاد

## 11) تصمیم‌هایی که قبل از پیاده‌سازی باید نهایی شوند

1. سایز پیش‌فرض صفحه اسناد: 20 یا 50؟
2. Export اسناد:
   - فقط صفحه جاری
   - کل نتایج فیلترشده
   - هر دو
3. مسیر نهایی PDF:
   - فقط ساختاریافته
   - یا نگه داشتن Legacy به عنوان fallback
4. سطح جستجوی عمیق:
   - همیشه فعال
   - یا اختیاری با toggle
